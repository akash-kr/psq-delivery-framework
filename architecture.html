Autonomous Delivery Framework

Owner: Akash · 2026-07-09 · builds on the Grounded Delivery Framework (v1)

You think, agents execute, machinery verifies. Everything below main is disposable; everything on main passed through you.

This document upgrades the existing contract → adapters → gates → proof skeleton from a process spec into an operating system: it adds the orchestration, steering, and enforcement layers the current repo lacks.

2L0 — Steering: “I only want to think”▾

The new ambition, so it gets the most design attention.

2.1 The Scoper

A dedicated agent role (a Claude Code skill) that turns a one-paragraph idea into an executable plan. It is the only agent you talk to at length.

Each issue the Scoper creates carries: a lane label, acceptance criteria as a checklist, the required proof that closes it, blocked-by relations, and an S/M/L size guess so you can sanity-check the plan in 30 seconds.

Sizing rule: every issue must be completable in one agent session and reviewable in under 10 minutes. If it isn't, split it. This single rule makes the rest of the system work.

2.2 Your decision surface

You interact with the system in exactly three places, nowhere else:

Plan approval

Approve or edit the Scoper's issue breakdown. Once per project or milestone.

Blocker answers

Agents post a Decision needed: comment and move the issue to Blocked. You answer; work resumes.

PR review

Proof bundle + preview link + small diff. Approve, or comment rework: to send it back with context.

Optional: a daily digest summarizing PRs awaiting you, blocked issues, what merged, what's in flight. Target steady state: ~15 min/day per active project.

3L1 — Control Plane: Linear▾

Linear is the source of truth for intent and state. The repo is the source of truth for code and proof. The daemon is the source of truth for dispatch.

3.1 State machine

  • Ready is the daemon's dispatch queue. Only unblocked Ready issues get agents.
  • Gate Check = agent finished, CI + eval agent running.
  • Human Review = PR open with proof bundle. A webhook syncs your merge → Done.
  • Priority ordering + blocked-by edges give the daemon its scheduling for free.

3.2 Lanes

Lane labels route output type exactly as lanes.yml defines: research produces cited docs, spec produces acceptance criteria, code produces PRs with tests, qa produces proof packages. The agent's workflow prompt branches on lane — the same machinery ships research memos and production code.

4L2 — Orchestrator: the Daemon▾

Conform to the Symphony spec (openai/symphony, Apr 2026) rather than inventing a scheduler. It's a small, well-specified loop: poll tracker → dispatch eligible issues to bounded-concurrency agent sessions → stream events → reconcile → retry with backoff → stop runs whose issues changed state. No database; tracker + filesystem are the recovery state.

Phase A
Mac mini

Daemon + 2–3 concurrent sessions. Always on, zero cloud cost, easy to observe (tmux / log tail).

Phase B
VM

Same daemon, more concurrency, 24/7 projects, Linux parity with production. Mac mini becomes your review/preview box or a second runner pool.

4.1 Workspaces without worktrees

Your instinct is right and Symphony agrees: no worktrees. Each issue gets a plain directory with a fresh shallow clone:

~/factory/work/<PROJECT>/<ISSUE-KEY>/
├── repo/   # fresh clone · branch feat/<ISSUE-KEY>
└── proof/  # gate logs · screenshots · eval reports
  • The branch is pushed to origin; the PR is the only artifact that matters.
  • The workspace is deleted when the issue reaches a terminal state (daemon cleanup hook).
  • Two agents never share a checkout → no branch juggling, and a destructive agent mistake is confined to a disposable clone. Merge conflicts are handled where they belong: at PR rebase time, by an agent told to rebase.

4.2 WORKFLOW.md

Per Symphony: each project repo owns a WORKFLOW.md — YAML front matter (states, concurrency, runner exe, timeouts, workspace hooks) plus the prompt body that tells the agent how to behave. Your WORKFLOW.template.md becomes this file. Process is code, reviewed like code.

5L3 — Runners: which agent executes▾

Primary
Claude Code headless

claude -p. Hooks give you enforceable guardrails — a PreToolUse hook can hard-block git push --force; a prompt can only ask nicely. Skills give reusable roles; subagents give cheap fan-out.

Secondary
Codex app-server

Symphony's native target — run the reference daemon unmodified while you build. Also a genuinely different model for the eval ring (§8).

Pluggable
Hermes

Wrap behind the same runner contract. Good candidate for narrow lanes (research, QA sweeps) where you control its harness fully.

Design rule: the daemon talks to runners through one thin contract — start(workspace, prompt) → event stream → exit status. Runner choice is a per-project or per-lane config value in WORKFLOW.md, not an architectural commitment.

Practical start: run the Symphony reference daemon with Codex to get moving this week, and make Claude Code the first runner you add — its hook system is what makes L4 enforceable rather than aspirational.

6L4 — Guardrails: prompted vs. enforced▾

The most important idea in this layer: a rule that only lives in a prompt is a suggestion. Split every rule into the tier that can actually hold it.

🔒 Enforced — machinery, agent cannot bypass

hooks · branch protection · CI · credentials
  • Nothing merges without you — branch protection on main: your review required, required status checks, no admin bypass, force-push disabled. This alone satisfies “nothing ever merged without me.”
  • No destructive git — PreToolUse hook + git wrapper blocks push --force, reset --hard, clean -fd, checkout to older commits, --amend, any push to main.
  • No secrets exposure — .env* never in the clone; daemon injects whitelisted env only; gitleaks scan as a required check.
  • Blast radius — deploy key scoped to one repo, push-only to feat/*; workspace is a disposable directory.
  • Spend caps — per-session timeout, daily session budget, token budget per issue.
  • Green-without-proof — CI fails the PR if the proof bundle is missing or invalid against the schema.

📜 Prompted — rules file, steipete-derived

AGENTS.md · WORKFLOW prompt body
  • Atomic commits — commit only files you touched, list paths explicitly (git commit -m "msg" -- path1 path2). One logical change per commit. Never amend.
  • Never delete files to silence lint/type errors; stop and record a blocker.
  • git status check before every commit; quote bracketed paths.
  • No unrelated refactors; no new paid dependencies without approval.
  • Mark uncertainty as assumptions; never claim done without proof.
  • End every run in the fixed Output / Proof / Decision / Next format.

Atomic commits matter beyond hygiene: they make your review fast (each commit tells one story) and make partial rejection possible (revert one commit, keep the rest).

7L5 — Harness: the per-issue loop▾

What one agent session does, start to finish:

1
READ

Issue + contract + WORKFLOW prompt + repo conventions.

2
G0 · SCOPE

Restate task, acceptance criteria, and proof plan as the first commit (docs/plan/<ISSUE-KEY>.md) — cheap to audit later.

3
BUILD

Smallest slices; atomic commit per slice; gates run per slice.

4
G1/G2 · GATES

Adapter commands: install / lint / typecheck / build / test — locally, every slice.

5
G3 · PROOF

collect-proof → gate logs, test output, screenshots (Playwright, desktop + mobile for UI), citations for research lanes.

6
G4 · PR

Fixed body format, proof bundle committed, preview link, “Decision needed,” recommended next issues.

7
HANDOFF

Linear → Human Review. Agent stops. Always.

Failure handling: a gate failing twice on the same cause → stop, write a blocker, move to Blocked. No thrash loops burning tokens against a wall — the retry budget is for transient failures, not for “try harder.”

Adapters keep this stack-agnostic: the loop never names npm or pip. detect-stack picks the adapter; the adapter maps G1/G2 to real commands. Adding WordPress or Go support = one small YAML file. The one-HTML-file case and the production-app case run the same loop.

8L6 — Evals: three rings, increasing cost▾

Ring 1 · Deterministic · cheapest
CI on the PR (GitHub Actions)

Install, lint, typecheck, build, tests, proof-bundle validation — on a clean machine, so the agent can't fake its environment. Required status check.

▼ passes
Ring 2 · Adversarial · cross-model
Eval agent reviews the PR

A different model than the builder (Codex checks Claude's work, or vice versa). Each acceptance criterion → met, with file/line/test cited as evidence. Diff scan for scope creep, deleted tests, hardcoded values pretending to be logic. UI lanes: screenshots compared against criteria. A fail sends the issue back to Ready with the verdict as context — before spending your attention.

▼ passes
Ring 3 · You · most expensive
Human review

Proof bundle + eval verdict + preview link + small diff. The system's job is to make this ring take under 10 minutes.

Cross-model review in Ring 2 is the cheap trick with outsized value: models are much better at finding flaws in work they didn't produce, and a different model family doesn't share the builder's blind spots.

Where Crabbox fits (honest take)

Crabbox = lease a fast cloud box, rsync your dirty checkout, run the suite remotely. It is not a review or proof tool — it's an execution substrate. For your system it's an optional Ring-1 accelerator: when a project's test suite gets slow, agents run crabbox run -- <gate> mid-loop instead of grinding the Mac mini, and CI stays the authoritative check. Skip it until a suite is slow enough to hurt. Your demo-verification need is served by Ring 2 + preview deploys, not by Crabbox.

9L7 — Proof Surface: demos you can click▾

Reviewing a diff is not reviewing a product. Every UI-bearing PR must carry a live preview:

  • Static / HTML → Cloudflare Pages or GitHub Pages per-branch preview (free, zero infra).
  • Node / Next / API → per-branch preview URL via docker-compose on the Mac mini (<issue-key>.preview.local behind Caddy) or a PaaS preview env (Vercel / Fly) per project.
  • Non-UI lanes → the proof bundle is the demo: test logs, cited research, before/after outputs.

The adapter's package/preview command owns this per stack. The PR body links it. You review product first, code second.

11Build Order — each phase useful on its own▾

Phase 0a weekend
Enforcement first

Branch protection on a real repo; guardrail hooks (git wrapper + PreToolUse); AGENTS.md v2 with atomic-commit rules; proof-validation CI job. Run the G0–G4 loop manually via Claude Code on 2–3 real issues. Validates the harness before any daemon.

Phase 1week 1
Linear + Scoper

Linear workspace with the state machine + lanes; build the Scoper as a Claude Code skill; scope one real project through it; execute issues manually. Validates the steering layer.

Phase 2weeks 2–3
Daemon on Mac mini

Deploy the Symphony reference implementation (Codex runner) pointed at Linear. Add the Claude Code runner behind the runner contract. WORKFLOW.md per project. Agents now work while you sleep.

Phase 3week 4
Eval ring + previews

Ring-2 eval agent as a GitHub Action on PRs; preview deploys per adapter; daily digest scheduled task. Review time drops to target.

Phase 4when needed
Scale

Move/extend daemon to the VM; Hermes as a lane runner; Crabbox if a suite gets slow; multiple projects concurrently.

Anti-goal at every phase: building orchestration before enforcement. A daemon that dispatches unguarded agents is a faster way to make messes.

12Open Questions — deliberately deferred▾

  • Linear ↔ GitHub sync — webhook service or polling? Phase 2 decision; polling is fine to start.
  • Per-project vs. shared daemon — start shared, split if projects need different trust levels.
  • Hermes runner contract — depends on what Hermes exposes; spec it when Phase 4 nears.
  • Multi-repo projects (e.g., Next + Strapi split repos) — contract spans repos; issue carries a repo target. Design when it first occurs.
  • Cost telemetry — Symphony tracks tokens per session; decide where you want to see it (digest vs. Linear comment).
