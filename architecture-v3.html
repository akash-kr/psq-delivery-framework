Architecture v3 · Merged · Draft for review

Agent Delivery Framework

Owner: Akash · 2026-07-09 · merges ARCHITECTURE.md (v2) + agent_delivery_harness_architecture.mdx (v0.1)

You think, agents execute, machinery verifies. Agent work always passes through isolated execution, policy checks, harness-created atomic commits, verification gates, evidence generation, and a draft PR — and nothing is ever merged without explicit human approval.

The core idea is not to build an agent that understands every stack. It is a repo-contract-driven harness that every repository teaches through configuration, rules, workflows, and evals — so the same machinery ships a single static HTML file, a WordPress site, a FastAPI app, a Node/React app, a Go service, or a full production application.

1Invariant & the Eight Layers▾

Agents do work; the harness owns delivery. Agents may edit files and propose implementation steps. The harness owns workspace setup, command permissions, path permissions, diff inspection, secret scanning, atomic commit creation, verification commands, evidence generation, PR creation, status checks, and review handoff.

The agent's final message is not proof. The harness-collected logs, diffs, commits, screenshots, artifacts, and status checks are proof.

The end-to-end flow through the layers:

2L0 — Steering: “I only want to think”▾

The MDX draft assumed a well-formed issue already exists. This layer is where it comes from — the mechanism that lets you run the system on ideas, not tickets.

2.1 The Scoper

A dedicated agent role (a skill, run interactively) that turns a one-paragraph idea into an executable plan. The only agent you talk to at length.

Each generated issue carries: a task class (ui / api / database / infra — drives eval requirements, §9), acceptance criteria as a checklist, the required proof that closes it, blocked-by relations, and an S/M/L size guess so you can sanity-check the plan in 30 seconds.

Sizing rule: every issue must be completable in one agent session and reviewable in under 10 minutes. If it isn't, split it. Example decomposition of "build a full production app": scaffold → data model → auth → core UI → API → tests → deployment preview → docs.

2.2 Your decision surface

You interact with the system in exactly three places:

Plan approval

Approve or edit the Scoper's issue breakdown. Once per project or milestone.

Blocker answers

Agents post a Decision needed: comment and move the issue to Needs Human Input. You answer; work resumes.

PR review

Proof packet + preview link + small diff. Approve, request changes, reject, split task, or mark unsafe.

Optional: a daily digest summarizing PRs awaiting you, blocked issues, what merged, what's in flight. Target steady state: ~15 min/day per active project.

3L1 — Control Plane: Linear + GitHub▾

Linear is the work queue and human-facing status surface. GitHub is the source of truth for code review and merge permissions. The daemon owns dispatch. The repo owns code, contracts, and proof.

3.1 Human-facing tracker states

A task becomes eligible for the harness when it has: status Ready for Agent, label agent, a repo reference (owner/repo), and acceptance criteria present. Large tasks are decomposed before implementation (§2). Priority ordering + blocked-by edges give the daemon its scheduling for free.

3.2 Lanes and task classes

Labels route output type: research produces cited docs, spec produces acceptance criteria, code produces PRs with tests, qa produces proof packages. Task classes (ui / api / database / infra) additionally select which eval gates are mandatory (§9.3).

3.3 The second state machine: harness run states

Tracker states are human-facing. The harness keeps its own operational state machine per run — this is what the daemon logs, retries, and recovers against:

CREATED → CLAIMED → PLANNING → PLAN_READY → PLAN_APPROVED | PLAN_SKIPPED
→ WORKSPACE_READY → AGENT_RUNNING → CHANGES_DETECTED → POLICY_CHECKING
→ COMMITTING → VERIFYING → EVIDENCE_BUILDING → PR_OPENED → REVIEWING
→ CHANGES_REQUESTED | ACCEPTED | FAILED | CANCELED

Keeping them separate means tracker ceremony never leaks into orchestration logic, and vice versa.

4L2 — Orchestrator & Workspaces▾

Conform to the Symphony spec (openai/symphony) rather than inventing a scheduler: poll the tracker on a fixed cadence → dispatch eligible issues to bounded-concurrency agent sessions → stream events → reconcile → retry with exponential backoff → stop runs whose issues changed state. No database required to start; tracker + filesystem are the recovery state.

Phase A
Mac mini

Daemon + 2–3 concurrent sessions. Always on, zero cloud cost, easy to observe (tmux / log tail).

Phase B
VM

Same daemon, more concurrency, 24/7 projects, Linux parity with production. Mac mini becomes the review/preview box or a second runner pool.

4.1 Workspaces: fresh clones, no worktrees

Both drafts agree: worktrees are powerful but operationally confusing once many agents, branches, and review loops are involved. Each task gets an isolated checkout and branch:

git clone --filter=blob:none git@github.com:org/repo.git WEB-123/repo
cd WEB-123/repo
git checkout main && git pull --ff-only
git checkout -b agent/WEB-123-short-slug
~/.agent-harness/
├── cache/
│   └── org-repo.git  # bare mirror — clones are local & fast; fetch base branch per run
└── runs/
    └── WEB-123/
        ├── repo/       # fresh clone · branch agent/WEB-123-slug
        ├── artifacts/  # screenshots/ · videos/ · logs/
        ├── proof.json
        └── run.log
  • The branch is pushed to origin; the PR is the only artifact that matters. Branch names require the issue ID (require_issue_id_in_branch).
  • The workspace is deleted when the issue reaches a terminal state (daemon cleanup hook).
  • Two agents never share a checkout — no branch juggling, and a destructive mistake is confined to a disposable clone. Merge conflicts are handled at PR rebase time, by an agent told to rebase.

4.2 WORKFLOW.md

Each project repo owns a WORKFLOW.md: YAML front matter (tracker config, concurrency, turn caps, review policy) plus the prompt body. Process is code, reviewed like code.

---
tracker:
  kind: linear
  project_slug: WEB
  required_labels: ["agent"]
  active_states: ["Ready for Agent", "Changes Requested"]
  terminal_states: ["Human Review", "Done", "Canceled"]
agent:
  max_concurrent_agents: 2
  max_turns: 8
review:
  handoff_state: "Human Review"
  require_draft_pr: true
  merge_policy: "human_only"
---
You are implementing Linear issue {{ issue.identifier }}.

1. Read the issue description and acceptance criteria.
2. Make the smallest correct change.
3. Do not modify unrelated files.
4. Run the required verification commands.
5. Produce a proof packet.
6. Open or update a draft PR.
7. Move the issue to Human Review only after verification passes.

5The Repo-Local Contract▾

This is what makes the harness application-agnostic. The harness doesn't know WordPress from FastAPI; each repo teaches it — how to install, run, test, build, produce demos, which files agents may touch, which commands are forbidden, and which outputs count as proof.

repo/
├── AGENTS.md              # human-readable agent rules
├── WORKFLOW.md            # orchestration config + prompt (§4.2)
├── .agent-harness.yaml    # machine-readable policy, harness-enforced
├── .eval/
│   ├── smoke.yaml
│   ├── acceptance.yaml    # task-class gate requirements (§9.3)
│   └── reviewer.md        # reviewer-agent instructions
└── .github/workflows/agent-gates.yml  # required PR status checks (§10.3)

5.1 AGENTS.md — the prompted rules

# Agent Rules
You may modify only files relevant to the assigned task.

Never edit: .env, .env.*, production secrets, deployment
credentials, branch protection settings.

Do not run: git reset --hard, git clean -fd, git push --force,
rm -rf outside the workspace, production deploy commands.

Do not merge. Do not close the task as Done.
Stop at Human Review with proof.

Plus the durable steipete-derived hygiene rules: never delete files to silence lint/type errors — stop and record a blocker; check git status before proposing commit groups; quote bracketed paths; no unrelated refactors; no new paid dependencies without approval; mark uncertainty as assumptions; never claim done without proof; end every run in the fixed Output / Proof / Decision / Next format.

Everything here is validated by the harness, not trusted to the agent:

6L3 — Drivers & Runners: two interfaces, not one▾

Separate who thinks from where commands run. This is what lets you swap Codex for Claude Code, or local execution for Crabbox, without touching anything else.

🧠 AgentDriver — who thinks

Codex App Server · Claude Code · OpenAI Agents SDK · Cursor CLI · Aider · Hermes / custom shell agent
  • startRun(input) → AsyncIterable<AgentEvent>
  • requestStop(runId)
  • resumeRun(runId, feedback) — powers the Changes-Requested loop

⚙️ RunnerProvider — where commands run

Local · Docker · Devcontainer · Crabbox · GitHub Actions · Cloud sandbox
  • prepareWorkspace(run) → Workspace
  • runCommand(spec) → CommandResult
  • collectArtifacts(patterns) → Artifact[]
  • dispose()

The harness supplies each agent: task description, acceptance criteria, repo rules, allowed and forbidden paths, verification requirements, output format, and stop condition. The harness collects from each agent: proposed plan, edited files, command requests, test results, commit-grouping proposal, risk notes, and a summary for the PR.

Start here
Codex App Server

Symphony's native target — run the reference daemon unmodified while everything else is built. Also the natural Ring-2 reviewer when Claude builds (§9).

Add first
Claude Code headless

claude -p with hooks — a PreToolUse hook can hard-block a forbidden command in-flight, complementing the policy wrapper. Skills give reusable roles (Scoper, Reviewer); subagents give cheap fan-out.

Pluggable
Hermes

Wrap behind AgentDriver. Good candidate for narrow lanes (research, QA sweeps) where you control its harness fully. Spec the contract when Phase 5 nears.

Runner choice is a per-project or per-lane config value in WORKFLOW.md, not an architectural commitment.

7L4 — Guardrails▾

The most important idea in this layer: a rule that only lives in a prompt is a suggestion. Guardrails are enforced in four places — identity permissions, git flow, command policy, and runtime isolation — with the prompted rules (§5.1) as the last, weakest tier.

7.1 Permission guardrails — the agent's identity

Use a restricted GitHub App or token. The agent identity structurally cannot: merge PRs, push to main, modify branch protection, edit repository secrets, approve its own PR, or deploy production changes.

GitHub App permissions:
  contents:        write, only to agent/* branches
  pull_requests:   write
  issues:          write
  checks:          write
  metadata:        read
  administration:  none
  secrets:         none
  deployments:     none by default

Branch protection on main — your review required, required status checks, force-push disabled, no admin bypass — is what makes "nothing ever merged without me" a property of the system rather than a hope.

7.2 Git guardrails — the harness owns git

7.3 Command guardrails — the policy wrapper

All shell commands flow through the policy engine: agent → command request → policy engine → allow / deny / require approval / sandbox. When Claude Code is the driver, its PreToolUse hook adds a second, in-flight enforcement point for the same policy.

✅ Allowed

  • git status · git diff
  • npm test · pnpm test · pytest
  • go test ./... · ruff check
  • npm run build

⚠️ Requires approval

  • Installing dependencies
  • Running Docker
  • Creating migrations
  • Changing lockfiles
  • Modifying infrastructure
  • Long-running services

⛔ Denied

  • Force push · hard reset
  • Deleting repo root
  • Editing env files
  • Production deploys
  • terraform apply · kubectl apply
  • Exfiltrating secrets

7.4 Runtime guardrails — isolation modes

Each run executes in one of: local-fresh-clone, container, devcontainer, Crabbox remote runner, cloud sandbox, or CI-only. For trusted personal repos, fresh clone + container is enough. For unknown repos or untrusted code, use stronger sandboxing. Blast radius is bounded either way: repo-scoped deploy key, push only to agent/*, disposable workspace.

7.5 Resource guardrails

Per-session timeout, max_turns per run, max_concurrent_agents, daily session budget, and token budget per issue — all daemon-level config, all from WORKFLOW.md.

8L5 — Harness: the Per-Issue Loop & Commit Manager▾

What one run does, start to finish. Note the division: steps 1–3 are the agent thinking; everything else is the harness.

1
READ agent

Issue + acceptance criteria + repo contract + WORKFLOW prompt + repo conventions.

2
PLAN agent

Restate task, criteria, and proof plan (docs/plan/<ISSUE-KEY>.md). For M+ tasks the plan gate can require your approval before implementation (PLAN_APPROVED vs PLAN_SKIPPED).

3
BUILD agent

Smallest correct change; no unrelated files; commands flow through the policy wrapper; gates run per slice for fast feedback.

4
POLICY CHECK harness

Change ledger computes the diff → reject forbidden paths → secret-scan → flag approval-required files (lockfiles, migrations).

5
COMMIT harness

Commit Manager creates atomic commits from validated groups (below).

6
VERIFY harness

Full gate suite via the detected stack: format / lint / test / build / smoke, per .agent-harness.yaml.

7
EVIDENCE harness

Build proof.json + artifacts: logs, screenshots (desktop + mobile for UI), citations for research lanes.

8
DRAFT PR + HANDOFF harness

Push branch, open draft PR with the fixed body format, link preview, move issue to Human Review. The run stops. Always.

Thrash rule: the same gate failing twice on the same cause → stop, write a blocker, move to Needs Human Input. The retry budget is for transient failures, not for "try harder."

8.1 Atomic Commit Manager — the algorithm

1.  Record baseline: base branch, base SHA, clean working tree
2.  Run the agent
3.  Compute changed files
4.  Reject forbidden changes (.env, secrets, prod infra, unrelated paths)
5.  Secret-scan the diff
6.  Ask the agent to propose commit groups
7.  Validate each group: files exist · no forbidden files ·
    no generated junk · one logical purpose per group
8.  For each group: unstage everything → stage explicit paths → commit
9.  Run final verification
10. Push the branch

Commit groups are structured data, not prose:

[
  {"message": "feat: add landing page hero",
   "paths": ["src/app/page.tsx", "src/components/Hero.tsx"]},
  {"message": "test: cover hero rendering",
   "paths": ["tests/hero.spec.ts"]}
]
git restore --staged :/
git add src/app/page.tsx src/components/Hero.tsx
git commit -m "feat: add landing page hero" -- src/app/page.tsx src/components/Hero.tsx
The harness stages explicit paths. It never relies on broad git add . for agent output.

Why this matters beyond hygiene: atomic commits make your review fast (each commit tells one story) and make partial rejection possible (revert one commit, keep the rest).

9L6 — Evals: did the agent actually do the work?▾

Ring 1 · Deterministic · cheapest
Automated checks on a clean machine (CI, required status)
  • Branch exists · PR opened · PR references the issue
  • Only allowed files modified · required commands passed
  • Artifacts exist · app starts · health checks pass
  • UI tasks include screenshots/video · PR includes the proof packet
▼ passes
Ring 2 · Adversarial · cross-model
Reviewer agent — a different model family than the builder

Codex reviews Claude's work, or vice versa: models find flaws far better in work they didn't produce. It inspects: does the diff match the task · unrelated changes · are the tests meaningful · missed edge cases · is the proof packet complete · would a human reviewer likely request changes. It may comment or fail a check — it must not approve or merge. A fail re-queues the issue with the verdict as context, before spending your attention.

▼ passes
Ring 3 · You · most expensive
Human review — the final gate

Actions: approve · request changes · reject · split task · mark unsafe. The system's job is to make this ring take under 10 minutes.

9.1 Acceptance-criteria evals

Each issue's criteria convert into checks wherever possible. UI example — criteria "hero section exists, CTA says 'Start free trial', works on mobile" become: a Playwright test for hero text, a Playwright test for CTA text, and screenshots at desktop + mobile widths. API example — criteria "endpoint returns profile, unauthorized → 401, response matches schema" become: unit tests for both cases, JSON-schema validation, and a smoke test against the local server.

9.2 Task-class gates — .eval/acceptance.yaml

task_classes:
  ui:
    required: [lint, build, screenshot_desktop, screenshot_mobile]
    optional: [playwright, lighthouse]
  api:
    required: [tests, schema_check, healthcheck]
    optional: [load_smoke]
  database:
    required: [migration_dry_run, rollback_plan, tests]
    require_human_approval: true
  infra:
    required: [plan_only, security_review]
    forbid_apply_without_human: true

9.3 The feedback flywheel

Human review outcomes are stored and become policy. Every recurring complaint turns into an automated check:

You saidThe system learns
"Agent changed unrelated CSS."Add stricter path-scope eval.
"Agent forgot the screenshot."Strengthen the UI evidence gate.
"Agent made one giant commit."Tighten atomic commit-group validation.

Repo-specific eval suites are versioned in .eval/, so the flywheel compounds per project.

10.2 PR body — the fixed handoff format

Every PR carries: issue link, branch, base SHA · summary · changed files · the atomic commit list with SHAs · verification results with ✅/❌ per gate · demo (screenshot path + preview URL) · risk notes · and the explicit line "This PR must be reviewed and merged by the repository owner."

10.3 Required status checks — agent-gates.yml

A minimal GitHub Actions workflow runs on every PR to main: verify the proof packet exists, reject forbidden env-file changes (git diff --name-only origin/main...HEAD | grep -E '^\.env' → fail), and list changed files. The harness publishes its own deeper agent/harness check on top. These are the required checks branch protection points at.

10.4 Clickable previews

Reviewing a diff is not reviewing a product. Every UI-bearing PR carries a live preview: static/HTML → Cloudflare Pages or GitHub Pages per-branch preview; Node/Next/API → per-branch preview URL via docker-compose on the Mac mini (<issue-key>.preview.local behind Caddy) or a PaaS preview env (Vercel/Fly). Non-UI lanes: the proof packet is the demo. You review product first, code second.

11Crabbox & Remote Execution▾

Crabbox (lease a fast cloud box, rsync the checkout, run the suite, stream output) slots in as a RunnerProvider — nothing else changes. It is a runner and evidence provider, not a policy system or review tool.

Useful for: remote test runs, browser checks, heavy builds, platform-specific validation, evidence-producing commands, artifact capture, and warm-box reuse. Timing: skip it until a suite is slow enough to hurt — plain GitHub Actions on the PR already gives the clean-machine guarantee, and your demo-verification need is served by Ring 2 + previews.

Crabbox runs commands and collects evidence. The harness governs safety, policy, delivery, and review. Do not treat Crabbox alone as the security boundary for hostile code without stronger isolation around it.

13Build Order — enforcement first, product later▾

Resolved sequencing: get the guarantees with existing tools before building custom harness software. A daemon dispatching unguarded agents is a faster way to make messes — and building a CLI product first contradicts "spend more time thinking."

The minimal end-to-end loop (what "working" means)

The first useful version does this reliably: pick one issue labeled agent → clone into an isolated run directory → create an agent branch → run the agent → reject forbidden changes → create atomic commits → run verification → generate the proof packet → open a draft PR → move the task to Human Review → prevent merge unless you approve.

15Open Questions — answer before implementation▾

Workspace & runtime

  • Should local runs be allowed, or should every run happen inside a container?
  • Which repos are trusted enough for local execution; which tasks require remote or stronger sandboxing?

Agent permissions

  • Should the agent be allowed to install dependencies with approval?
  • Should the agent be allowed to start long-running services?

Verification

  • Which gates are always required vs. stack-specific?
  • What counts as proof for UI work; what counts for backend work?

Review

  • Should PRs always be drafts? Should reviewer-agent comments block review?
  • Should missing screenshots block UI PRs? Should dependency changes always require manual approval?

Scaling & integration

  • How many concurrent runs are acceptable? Should agent attempts be capped per issue? Should failed attempts create follow-up issues? Should large issues be auto-split?
  • Linear ↔ GitHub sync: webhook service or polling? (Polling is fine to start.)
  • Per-project vs. shared daemon — start shared, split on differing trust levels.
  • Multi-repo projects (e.g., Next + Strapi split) — contract spans repos; issue carries a repo target. Design at first occurrence.
  • Cost telemetry — tokens per session surface in the digest or as a Linear comment?
