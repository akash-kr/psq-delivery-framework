Proof, Automated

QA Automation

QA spends 90% defining the test cases and expected outcomes, and 10% verifying — everything in between is automated. Each rule becomes a test that runs on its own, with a screenshot as evidence.
1

DEFINE — 90%

One test per contract rule, in plain words — Given / When / Then. This is where QA's judgement goes.

2

AUTOMATE

Each test becomes a Playwright script that clicks through the real UI — runs on every code push, no one watching.

3

VERIFY — 10%

Playwright screenshots every screen and tags it PASS or FAIL vs the design. QA just reviews the flags.

Speed, With Guardrails

How Dev ships with AI

AI does the heavy lifting; the pipeline enforces quality. Clear ownership, no chasing.
  • Claude Code builds UI straight from the contract — and writes the unit tests with it.
  • Ownership is clear: Dev owns unit tests (the logic). QA owns Playwright (the user journey).
  • Midscene.js + Playwright — AI, vision-driven tests written in plain English. It reads the screen like a human, so there are no brittle selectors to chase when the UI changes.
  • Every merge request gets an AI review (Codex) before a human looks — catches the obvious first.
  • GitLab runs everything on each push: unit + Playwright + screenshots. No merge unless all green.
Every push, in GitLab
Commit · dev pushes code
AI review · Codex checks the diff
CI runs · unit · Playwright / Midscene · screenshots
Merge — only if green ✓
The pipeline is the gate. Quality is enforced by the tools, not by managing people.

Automated Escalations

When something's stuck, the system shouts

A blocked gate — or a contract edited without sign-off — holds the milestone and the invoice. A watcher runs on a schedule, checks for these, and pings the right person automatically. It also keeps an eye on contract.yml so any unapproved change is caught the moment it lands.
watcher.rules.yml runs every 30m
# escalate when… gate_red: if: status == red for: > 4h notify: owner, #dev-qa contract_change: watch: contract.yml if: merged && approvals < 2 notify: design, qa milestone_risk: if: due_in < 24h && gates != all_green notify: owner, me
One file. Change a threshold, change the rule — no code.
Built for Dev & QA: they see a blocker the moment it matters, not days later. No one has to remember to chase it.
